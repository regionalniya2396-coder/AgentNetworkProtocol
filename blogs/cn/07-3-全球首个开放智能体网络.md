想象一下未来的某一天：

你只需要对 AI 助手说一句——
**“帮我规划下周去上海的行程。”**

它不会打开五个 App，也不会让你再去注册账号。
它会在后台直接和**天气智能体**确认晴朗的周末、与**酒店智能体**锁定外滩江景房、与**高铁智能体**完成购票。

几秒钟后，你的行程自动全部安排完毕。

这并不是科幻，这是今天正在发生的现实。
这是**开放智能体网络（Open Agent Network）**带来的互联网新形态。

过去一年，全世界都在投入构建“超级助手”：ChatGPT、Gemini、Kimi、豆包、夸克、千问、灵光……
AI 大脑越来越聪明，但它们却被困在一个个**无法互通的孤岛**里：

* 不能跨平台调用数据
* 不能打通商业服务
* 不能与其它 AI 协作

AI 很强，但互联网太弱。

于是，我们开发了一个智能体协议，并且基于协议，开始构建一张新的网络——
**让所有智能体都能自由连接、互操作、互信任的开放网络。**
就像 HTTP 连接网页一样，这张网络将连接全世界的智能体。

# **什么是开放智能体网络？一句话讲清楚**

**开放智能体网络（Open Agent Network）= 一套协议 + 一张网络，让任何智能体都能跨域连接彼此。**

简单说：

* **开发者**：只需一次开发，你的智能体就能连接全球所有服务
* **用户**：只需一个助手，就能访问整个互联网的服务
* **企业**：只需一次部署，企业的智能体就可以服务全球所有的用户

这是“AI互联网”的雏形。

---

# **为什么 AI 需要这样一张网络？三个根本性痛点**

**1. AI 天生擅长理解，但不擅长连接**

当前 AI 能听懂你的意思，却无法直接访问互联网的真实服务（酒店、天气、交通、教育……）。

它是超强大脑，却缺乏“手脚”。

**2. 服务被封在 App 和平台里**

旅⾏、支付、医疗、教育……都被切割在不同 App、不同平台、不同公司。

AI 想调用，必须跳转、注册、反爬、私有 API……
根本无法自动化。

**3. AI 之间不能互相协作**

一个助手帮你查酒店，但不能找另一个助手约会议，
不能和朋友的助手自动协调聚会，
不能和企业客服智能体交接。

孤岛式的 AI，正在限制AI能力的发挥。

# **开放智能体网络的三层架构**

开放智能体网络由三层组成，目标只有一个：
让需求侧智能体能自由连接供给侧智能体。

![三层架构](/blogs/images/agent-network/network-architecture.png)

1. 需求侧（Demand Side）
包括个人助手、垂直助手、AI 应用。
他们负责理解用户意图，并向网络发起请求。
最终实现：**一个入口，连向全网。**  

2. 连接层（Connection Layer）
智能体的“导航 / 搜索 / 信任”基础设施。
负责让智能体彼此可发现、可理解、可信任、可连接。
他们不是平台，而是**协议驱动的连接层**。

3. 供给侧（Supply Side）
提供真实服务的智能体：酒店、餐饮、教育、娱乐……
最终实现：**一次接入，全网可用。**

# **为什么我们坚持“协议”，而不是“平台”？**

很多公司在做 AI 平台、AI 应用商店、AI 框架……
但平台永远解决不了“互联互通”的问题。

## 平台 = 封闭的花园

* 你必须在它的规则下开发
* 必须使用它的 SDK
* 必须将用户和数据绑定在它的系统中

## 协议 = 开放的公路

* 你想怎么实现都可以
* Python、JavaScript、Java 全部支持
* 本地部署、云部署都能互通
* 没有平台锁定、没有技术依赖、没有权限限制

HTTP 成就了网页互联网。
SMTP 成就了全球邮件系统。
RSS 成就了信息订阅网络。

**ANP（Agent Network Protocol）将成就“智能体互联网”。**

---

# **为什么首选ANP，而不是其他协议？**

ANP 是全球首个专为**互联网协作**而生的开源协议。

**它与其他协议最大的不同，在于“身份”。**

我们采用了 **DID（分布式身份）** 技术。这让智能体拥有了像**电子邮件（Email）**一样的自由：
*   **打破围墙**：无论对方使用什么服务商，你们都能直接对话。
*   **互联互通**：不再需要重复注册账号，一个 ID 走遍全网。

其他协议大多没有解决智能体之间身份互通的问题，而 ANP 让智能体真正走进了**互联网**。

当然，未来我们希望只要一个智能体支持任何一个协议，无论MCP、A2A、ANP，都可以接入到网络中，并且互相协作。

# 怎么接入智能体网络中

我们提供两种接入方式，一种ANP原生接入，一种MCP-to-ANP桥接接入。

**ANP原生接入**：直接使用ANP SDK接入，能够支持完整的双向通信能力，功能更加丰富。

**MCP-to-ANP桥接接入**：对于已经实现了MCP的智能体，我们提供了MCP-to-ANP的桥接方案。只要你的智能体支持MCP，通过桥接就可以一键接入智能体网络，无需修改代码。
